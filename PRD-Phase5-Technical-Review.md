# 第一部分：当前 PRD 的根本问题

## 问题 1：目标产品错位

原 PRD 的 Mission 写明：

> "成为项目未来所有能力（Agent、Workflow、Knowledge、MCP、Plugin）的统一上下文基础设施"

**这是一个平台型基础设施的愿景，不是视觉对话助手的需求。**

把 LangGraph 的 Checkpointer+Store、Mem0 的记忆管线、Letta 的 OS 比喻、CrewAI 的 Agent 级记忆——所有这些的设计目标都是**多 Agent、长任务、知识密集型**场景——塞进一个摄像头+麦克风的产品，就像给自行车装航空发动机。

### 真实产品 vs PRD 假设的产品

| 维度 | 真实产品 | PRD 假设的产品 |
|------|---------|---------------|
| 交互模式 | 按住说话，松手听回复，看画面 | 多轮深度对话 |
| 单次交互时长 | 5-30 秒 | 分钟级 |
| 对话轮次 | 1-5 轮 | 8-20+ 轮 |
| 核心价值 | 即时的视觉理解+语音交互 | 知识积累、项目协作 |
| 记忆需求 | 认识你、记住偏好 | 记住一切、建立知识图谱 |
| 用户预期 | 工具 | 伙伴/Agent |

**结论：PRD 设计的是"伙伴/Agent"的记忆系统，但这个产品目前的形态是"工具"。**

---

## 问题 2：Memory 类型堆砌——没有做减法

PRD 中的 L3 层定义了七种记忆：

```js
profile       // 个人信息
preferences   // 偏好
facts         // 事实
projects      // 项目
recentTopics  // 话题
decisions     // 决策
knowledge     // 知识
```

**逐条审视**：

| 记忆类型 | 视觉助手需要？ | 理由 |
|----------|:--:|------|
| profile | ✅ 需要 | 姓名、学校等基本信息有助个性化 |
| preferences | ✅ 需要 | 回复风格、语言偏好直接影响体验 |
| facts | ⚠️ 弱需要 | 对视觉场景帮助有限——"用户在学习 Python"不会帮你识别一棵树 |
| projects | ❌ 不需要 | 这是 IDE/工作流产品的需求。视觉助手没有"项目"概念 |
| recentTopics | ❌ 不需要 | ChatGPT 级别的功能。对 30 秒交互模式无意义 |
| decisions | ❌ 不需要 | 视觉对话不做技术决策追踪 |
| knowledge | ⚠️ 弱需要 | 模糊概念。对视觉识别无帮助 |

**真正需要的只有 2 种：Profile + Preferences。** 其余 5 种是直接搬运通用 Agent 框架的概念。

---

## 问题 3：Conversation 为重的架构——对视觉产品是错的

原 PRD 架构的核心是 Conversation：

```
Conversation Manager → Session CRUD → Message CRUD → Compression → Search → Export
```

但视觉 AI 助手的核心交互单元不是"对话"，是**"视觉时刻"**。

用户不会在视觉助手里：
- 导出对话为 Markdown
- 搜索历史对话中的关键词
- 给对话打标签分类
- 归档旧对话

用户会做的事：
- 拿起手机对着花问"这是什么？"
- 对着电脑屏幕问"这个报错怎么修？"
- 对着菜单问"这个菜辣不辣？"

**这些交互的共同特征：视觉是短暂的，查询是独立的，上下文是"现在看到的东西"。**

回顾交互历史的场景极少——用户不会"回顾上次识别花的那次对话"。这跟 ChatGPT 完全不同。

---

## 问题 4：检索架构过度设计

原 PRD 的 Retrieval Layer：

```
L1 检索 → L2 检索 → L3 检索：
  文本匹配 → 频率加权 → 置信度过滤 → Top-K 截断
  → Phase 5.2 升级：Transformers.js embedding → 余弦相似度
  → BM25 混合搜索 → Rerank
```

**对于 200 条记忆事实、每次注入最多 5 条的场景，这个管线是过度设计。**

真实场景下：
- 用户说"看看这个"
- AI 需要从记忆中检索相关内容
- 200 条文本事实，其中可能只有 2-3 条与当前视觉场景相关
- 简单文本匹配已经足够（如果记忆设计得足够精简）

**Transformers.js 的 WASM 模型最小也要 20MB+，加载时间 2-5 秒。为了 200 条记忆做语义搜索？ROI 极低。**

---

## 问题 5：上下文压缩——解决了不存在的问题

原 PRD 花大量篇幅设计上下文压缩：

> 当 messageCount > 16（8轮），触发压缩。锚定迭代式摘要。结构化摘要（目标/决策/文件/偏好）。

**问题是：视觉对话助手会有 8 轮对话吗？**

实测场景：
- 识别物体：1-2 轮
- 翻译文字：1 轮
- 询问建议：2-3 轮
- 最长场景（调试/教学）：5-10 轮

8 轮对于这个产品来说是**边缘场景**，不是一个需要专用压缩架构的核心问题。而且视觉场景中，最关键的信息是**画面本身**——画面不能被压缩成文本摘要。

当前的 6-8 轮滑动窗口（Phase 4 的能力）已经足够应对 95% 的使用场景。

---

## 问题 6：Hook 和 Plugin 系统——现在设计太早

原 PRD 设计了完整的 Hook 系统和 Plugin 接口。**在产品的功能集还没验证之前设计扩展系统，是本末倒置。**

软件工程第一原则：**先让核心功能稳定，再考虑扩展性。** 过早抽象是万恶之源。

# 第二部分：视觉 AI 助手的真实记忆需求

## 核心洞察

**视觉 AI 助手的记忆不是"对话记忆"，而是"视觉上下文记忆"。**

用户和这个产品的每一次交互，都有一个**图像帧**。这是所有上下文中信息密度最高的部分。当前 PRD 把帧当作对话历史的一个字段来存储，但帧自身应该是记忆体系的核心。

## 真实高频需求

| 需求 | 频率 | 当前 PRD 覆盖？ |
|------|:--:|:--:|
| 记住"我是谁" | 每次 | ✅ profile |
| 记住"我偏好什么风格" | 每次 | ✅ preferences |
| 识别常见物体时更快/更准 | 中 | ❌ **缺失** |
| 记住经常看的物体/场景 | 中 | ❌ **缺失** |
| 跨场景的上下文感知（从室内到室外等） | 低 | ❌ **缺失** |
| 学习用户的视觉领域（医学影像？机械零件？植物？） | 中 | ❌ **缺失** |

**发现：当前 PRD 覆盖了 2/6 的高频需求。缺失的 4 个都是视觉相关的。**

## 视觉记忆 vs 文本记忆

| 文本记忆（PRD 当前设计） | 视觉记忆（缺失） |
|--------------------------|-------------------|
| "用户在学习 Python" | 用户经常看 PCB 电路板 |
| "用户偏好简洁回答" | 用户经常在户外自然光下使用 |
| "用户选了方案 B" | 用户常看的物体类型：植物 > 电子设备 > 食物 |
| "用户在中北大学" | 用户的典型使用场景：桌面（60%）、户外（30%）、室内暗光（10%） |

**文本记忆对视觉对话的实际帮助有限。** 知道"用户大一"不会帮助 AI 更好地识别摄像头里的电路板。但知道"用户经常看电路板"会让 AI 更主动地提供电子工程相关的上下文。

# 第三部分：重新设计——视觉优先记忆架构

## 原则

1. **视觉优先**：记忆的核心单元是视觉场景，不是文本对话
2. **极简分层**：不是 3 层，不是 7 种类型
3. **零检索开销**：记忆足够少，全量注入而非检索
4. **本地零依赖**：不使用任何外部向量库、图数据库、embedding 模型

## 推荐架构：Visual Context Memory (VCM)

```
┌──────────────────────────────────────────┐
│          VISUAL CONTEXT MEMORY            │
├──────────────────┬───────────────────────┤
│  Session Context │  Personal Context     │
│  (本次会话)      │  (跨会话持久)         │
├──────────────────┼───────────────────────┤
│ • 当前图像帧     │ • 用户画像 (~10条)    │
│ • 最近 3 轮对话  │ • 偏好设置 (~5条)     │
│ • 当前场景特征   │ • 视觉领域标签 (~5条) │
│   (亮度/时间/?)  │ • 常用场景快照 (~3条) │
│                  │                       │
│ 生命周期: 会话级 │ 生命周期: 永久        │
│ 存储: 内存       │ 存储: localStorage    │
│ 容量: ~5KB       │ 容量: ~2KB            │
│ 注入: 直接       │ 注入: System Prompt   │
└──────────────────┴───────────────────────┘
```

### 为什么是两层而不是三层？

原 PRD 的 L1（内存 8 轮）+ L2（localStorage 完整历史）+ L3（localStorage 提取事实）。

**L2 是冗余的。** 对于视觉助手，完整对话历史的实用价值极低。用户几乎不会回顾历史。如果真的需要，文本搜索当前对话即可。

**合并后的两层**：
- Session Context：当前交互的实时上下文
- Personal Context：跨会话的累积认知

没有"对话管理"层，因为没有需要管理的多对话场景。**用户和视觉助手之间只有一个持续的"关系"，而非多个"项目"。**

---

## Personal Context 精确数据模型

```js
{
  version: 1,
  updatedAt: 1718300000000,

  // ── 我是谁 ──
  profile: {
    name: null,          // "小明"
    role: null,          // "大一学生, 中北大学, 智能测控工程"
    language: "zh-CN",
  },

  // ── 我怎么用 ──
  preferences: {
    responseStyle: "简洁",    // "简洁" | "详细" | "口语化"
    responseLength: "短",     // "短"(~50字) | "中"(~100字) | "长"(~150字)
    technicalLevel: "intermediate",
  },

  // ── 我看什么 ──  (视觉特有)
  visualDomains: [
    { domain: "电子产品", weight: 0.8, lastSeen: 1718300000000 },
    { domain: "植物", weight: 0.5, lastSeen: 1718300000100 },
    // 从对话中推断：用户频繁问某个领域的问题
  ],

  // ── 我在哪用 ──
  usagePatterns: {
    typicalLighting: "indoor",    // "bright_outdoor" | "indoor" | "dim"
    typicalDistance: "close",     // "close" | "mid" | "far"
    // 从帧质量推断
  },

  // ── AI 对我的认知 ──
  aiNotes: [
    // 自由文本，AI 自动维护
    "用户经常在桌面上使用，可能是在宿舍/实验室",
    "用户对电子元件感兴趣，可能是专业课相关",
    "用户偏好一句话回答而非段落",
  ],
}
```

### 关键设计决策

**1. `aiNotes` 替代 `facts` 数组**

原 PRD 的结构化 facts（`{id, type, content, confidence, source}`）过度工程化。对于 < 50 条信息的总量，结构化存储带来的复杂性远超收益。

`aiNotes` 是一个自由文本数组，AI 自己维护：
- AI 在对话后追加新认知
- AI 在发现矛盾时替换旧认知
- 注入到 System Prompt 时直接拼接
- 用户可以直接编辑文本

**这是一种"Claude CLAUDE.md"模式的微型化应用。简单 > 结构化。**

**2. `visualDomains` 是视觉特有的记忆**

这是当前 PRD 完全缺失的维度。视觉助手可以通过多轮对话推断用户的使用领域：
- 用户频繁问"这是什么植物？" → domain: 植物
- 用户频繁对着电路板提问 → domain: 电子产品
- 用户频繁翻译文字 → domain: 文档/文字

当 AI 知道用户的主要视觉领域后，可以在 System Prompt 中注入：
- "用户经常识别植物，如果画面中有植物，优先从植物学角度回答"
- "用户经常看电子产品，回答时可以使用专业术语"

**这是视觉 AI 独有的价值——通过学习"你在看什么"来提升后续视觉理解的质量。**

**3. `usagePatterns` 从帧元数据中自动学习**

不依赖用户告知。从图像帧的像素分析中自动推断：
- 亮度分布 → 典型使用环境（室内/室外/暗光）
- 边缘密度 → 典型观察距离（近景/中景/远景）
- 帧中主要颜色温 → 场景类型

这些信息可以帮助 AI 调整回答策略（如暗光时提示开灯，或针对远景优化描述）。

---

## 注入策略：全量注入，零检索

Personal Context 的数据量约 2KB（~500 tokens）。直接全部注入到每条 System Prompt 的开头：

```
System:
你是视觉对话助手...

[关于用户]
- 小明，大一学生，中北大学，智能测控工程专业
- 偏好简洁回答，一句话
- 技术基础：中级
- 经常看的领域：电子产品、植物
- 通常在室内桌面使用，中等距离
- AI 对他的认知：对电子元件感兴趣，可能是专业课相关

[当前对话]
...
```

**为什么全量注入优于检索注入？**

| 维度 | 全量注入 | 检索注入 |
|------|---------|---------|
| 总数据量 | ~500 tokens | ~500 tokens（需包含检索开销） |
| 额外延迟 | 0 | 检索计算 + API 调用 |
| 正确性 | 100%（无遗漏） | < 100%（检索可能漏） |
| 实现复杂度 | 简单字符串拼接 | 文本匹配/向量搜索/rerank |
| Token 成本 | ~500 tokens/次 | ~500 tokens/次（近似相同） |

**当数据量小于上下文窗口的 0.5% 时，检索是多余的。** 这正是 Personal Context 的场景。

# 第四部分：技术可行性与 ROI

## 必须做 (P0)

| 能力 | 工作量 | 用户价值 | 技术复杂度 |
|------|:--:|:--:|:--:|
| **对话持久化** (F1) | 低 | 极高 | 低 — 每条消息后 `localStorage.setItem()` |
| **Personal Context** (画像+偏好+视觉领域) | 低 | 极高 | 低 — ~200 行 JS，零后端改动 |
| **Personal Context 注入** | 低 | 高 | 低 — 修改 `buildMessages()` 追加前缀 |

**三个能力总工作量：~1 天。ROI：极高。**

## 建议做 (P1)

| 能力 | 工作量 | 用户价值 | 技术复杂度 |
|------|:--:|:--:|:--:|
| **AI 自动更新 Personal Context** | 中 | 高 | 中 — 新后端端点，异步调用 |
| **视觉领域自动推断** | 中 | 中 | 中 — 从对话文本中推断领域 |
| **Personal Context 查看/编辑 UI** | 低 | 中 | 低 — 设置面板新标签页 |

## 不建议做 (当前 PRD 中的过度设计)

| 能力 | 理由 |
|------|------|
| **多会话管理** (F2) | 视觉助手不需要。用户和设备之间是一个持续的关系 |
| **上下文压缩** (F5) | 解决不存在的问题（视觉助手几乎不会有 8+ 轮对话） |
| **对话搜索** (F6) | 低频需求。未来可加，MVP 不需要 |
| **对话标签/归档** (F8) | 项目管理功能，视觉助手不需要 |
| **语义搜索** (F13) | 500 tokens 的记忆不需要语义搜索 |
| **Hook/Plugin 系统** | 核心功能未验证前，过早抽象 |
| **Vercel KV 备份** (F15) | 数据量太小（< 100KB），不需要云端 |
| **知识图谱** (Phase 6) | 对视觉助手无意义 |
| **多 Agent 共享记忆** | 产品不需要 |

# 第五部分：创新方案 — Vision-Centric Memory

## 当前缺失：视觉特征记忆

如果要做视觉 AI 助手，最有价值的记忆是**视觉记忆**本身。但当前 PRD 完全没有涉及。

### 概念：Visual Fingerprint Memory

不存储原始图像（隐私+存储成本），而是存储**视觉特征指纹**：

```
每次交互:
  1. 从帧中提取轻量视觉特征（无需神经网络，纯像素统计）
     - 主色调分布（HSV 直方图，5 bins × 3 channels = 15 floats）
     - 亮度分布（均值 + 标准差）
     - 边缘密度（Sobel 梯度幅值均值）
     - 纹理复杂度（局部熵均值）
  2. 与已有的"场景快照"比对
  3. 如果匹配已知场景 → 加载该场景的上下文
  4. 如果是新场景 → 记录新快照
```

**总存储成本：每个快照 ~100 bytes。100 个快照 = 10KB。**

### 价值场景

| 场景 | 无视觉记忆 | 有视觉记忆 |
|------|-----------|-----------|
| 用户常在桌面看电路板 | 每次 AI 都从零开始 | AI 识别到"这是之前看电路板的场景"，主动提供电路相关知识 |
| 用户在户外看植物 | AI 通用回答 | AI 知道用户在户外，提示"可能需要更近距离拍摄" |
| 用户在不同光照下使用 | AI 无感知 | AI 知道当前比平时暗，提示"是否需要闪光灯？" |

### 是否该现在做？

**不。** 这属于 Phase 6 的创新方向。MVP 应该从最简单的文字记忆开始，验证用户价值后再迭代到视觉记忆。

但 PRD 中完全缺失这个方向是一个战略失误——当未来想扩展时，当前的数据模型没有为视觉特征预留任何空间。

# 第六部分：重新定义 Phase 5

## Phase 5 的真正目标

> **从"一次性工具"到"认识你的工具"。**
>
> 不是让 AI 记住你的全部对话历史。
> 而是让 AI 认识你——你是谁、你怎么用、你在看什么。
> 每次打开，AI 不需要你重新介绍自己。

## MVP 范围（砍掉原 PRD 的一半）

| # | 功能 | 对应原 PRD | 变化 |
|---|------|:--:|------|
| M1 | **对话持久化** | F1 | 保留，简化。仅保存当前对话，不支持多会话 |
| M2 | **Personal Context 系统** | F3+F4 合并+重塑 | 替代原 L3 记忆体系。更简单、更聚焦 |
| M3 | **AI 自动更新 Personal Context** | F3 的一部分 | 简化。异步更新，自由文本格式 |
| M4 | **Personal Context 注入** | 新增 | 每次请求注入到 System Prompt |
| M5 | **Personal Context 查看/编辑** | F4 简化版 | 设置面板中的一个标签页 |

**砍掉**：F2（多会话管理）、F5（上下文压缩）、F6（对话搜索）、F7-F15（所有后续阶段功能）。

**总工作量：原 PRD 的 ~30%。用户价值：原 PRD 的 ~90%。**

# 第七部分：最终评估与建议

## 当前 PRD 的技术评分

| 维度 | 评分 | 说明 |
|------|:--:|------|
| 行业调研 | 9/10 | 深入全面 |
| 方案适配性 | 4/10 | 为通用 Agent 设计，非视觉助手 |
| 复杂度控制 | 3/10 | 严重过度工程化 |
| 视觉创新性 | 2/10 | 几乎为零 |
| 技术可行性 | 6/10 | 可实现，成本过高 |
| 文档质量 | 8/10 | 结构清晰，详实 |
| **综合** | **5/10** | 方向对（需要记忆），方案错（用错范式） |

## 主要风险

| # | 风险 | 严重度 |
|---|------|:--:|
| 1 | 7 种记忆类型 + 3 层架构会让 Agent 实现出过度复杂的代码，维护成本高 | 高 |
| 2 | 多会话管理对视觉产品无价值，增加 UI 复杂度，稀释核心体验 | 高 |
| 3 | 上下文压缩解决不存在的问题（视觉助手罕见 8+ 轮对话） | 中 |
| 4 | Transformers.js 语义搜索对 < 500 tokens 的记忆是浪费 | 中 |
| 5 | Hook/Plugin 系统过早设计，会成为代码债务 | 中 |

## 过度设计清单

1. **L2 中期记忆层**：对视觉产品无价值
2. **facts/decisions/projects/recentTopics/knowledge 五种记忆类型**：仅 profile + preferences 有用
3. **上下文压缩锚定迭代摘要**：过度方案解决边缘问题
4. **对话搜索**：低频需求，MVP 不应包含
5. **Transformers.js 语义搜索**：对 < 500 tokens 的记忆无价值
6. **Hook/Plugin 系统**：过早抽象
7. **Vercel KV 云端备份**：数据 < 100KB，不需要
8. **记忆导出/导入/共享**：MVP 不需要
9. **对话标签/归档**：项目管理功能

## 缺失清单

1. **视觉领域记忆 (visualDomains)**：完全缺失。这是视觉助手独有的价值
2. **使用场景感知 (usagePatterns)**：缺失。对优化回复质量有帮助
3. **视觉特征记忆 (Visual Fingerprint)**：长期战略缺失。当前数据模型未预留空间
4. **AI 自由文本笔记 (aiNotes)**：比结构化 facts 更适合数据量 < 50 的场景
5. **帧元数据提取**：从图像像素中自动学习使用场景，而非依赖用户告知

## 推荐的 Phase 5 技术路线图

```
Phase 5 MVP (1-2 天)
├── M1: 对话持久化 (localStorage, 仅当前对话)
├── M2: Personal Context 数据模型
├── M4: Personal Context System Prompt 注入
├── M5: Personal Context 查看/编辑 UI
└── M3: 简单规则提取 (可选, 关键词匹配)

Phase 5.1 (验证后)
├── M3 升级: LLM 驱动的 Personal Context 更新
└── 对话历史搜索 (纯文本匹配)

Phase 5.2 (产品验证后)
├── visualDomains 自动推断
├── usagePatterns 从帧元数据学习
└── 视觉特征指纹 (场景匹配)

Phase 6 (长期)
├── 多设备 Personal Context 同步 (Vercel KV)
├── 图像相似度记忆 (轻量视觉 embedding)
└── 跨会话视觉场景连续性
```

## 未来 2-3 年最具创新性的方向

1. **Visual Fingerprint Memory**：不存储图像，存储视觉特征指纹。让 AI 在看到相似场景时自动加载相关上下文。这是视觉助手独有的能力，文本 Agent 做不到。

2. **Domain-Adaptive Vision**：通过 visualDomains 的学习，AI 逐渐调整视觉理解策略——如果用户是医学生经常看医学影像，AI 应该调整到更专业、更准确的医学视觉理解模式（通过 System Prompt 引导 VLM）。

3. **Passive Context Awareness**：不是等用户说话才工作。通过持续分析帧的低维特征（亮度、运动、颜色变化），AI 感知场景切换（从桌面到户外、从白天到黑夜），自动调整回答策略。

4. **Memory as a Product Feature**："AI 认识你"本身就是卖点。不是后台功能，是前台体验。首次使用时 AI 问几个问题建立初始 Personal Context，之后每次打开都感觉"AI 记得我"。

## 结语

原 PRD 是一个**优秀的通用 Agent 记忆系统设计**——如果目标产品是 LangGraph Agent、CrewAI 多 Agent 系统、或 ChatGPT 级别的对话产品。但对于 OmniDeskAI（视觉 AI 对话助手），它是**用错了范式**。

核心错误：**把"记忆"理解为"对话历史的持久化和检索"，而不是"对用户的理解和认知"。**

对于视觉助手，正确的记忆体系是：
- **少**（< 50 条信息，全量注入）
- **视觉感知**（学习用户在什么场景、看什么类型的物体）
- **轻**（~500 tokens，零检索开销，纯文本）
- **自动**（AI 自己维护，用户只需偶尔审查）

> **"Perfection is achieved not when there is nothing more to add, but when there is nothing left to take away."**
> — Antoine de Saint-Exupéry

**从这个产品中拿掉的记忆功能，比加上的更有价值。**
